# 컨테이너 startup 시 DB 스키마 자동 적용

## 배경 / 문제

운영 초기 배포에서 **DB는 생성됐지만 스키마(테이블)가 만들어지지 않아** 프론트엔드가
백엔드 호출에 실패(`ECONNREFUSED` / 기동 레이스)하는 사고 발생.

원인:
- 배포 이미지 CMD는 `uvicorn`뿐 — 마이그레이션(`alembic upgrade head`) 단계가 없음.
- `app/main.py` lifespan은 `bootstrap_admin()`만 호출하고, 스키마가 없으면
  `try/except`로 조용히 스킵 → 테이블이 없는 채로 API만 기동.
- 결과적으로 매 신규 배포마다 수동 `alembic upgrade head`가 필요했음.

## 목표

컨테이너 기동 시 스키마가 없으면 **자동 생성**, 이미 있으면 **무충돌 no-op**.
멱등성은 alembic 버전 테이블(`alembic_version`)로 보장 — 누락된 리비전만 적용하므로
"already exists" 충돌이 발생하지 않는다. (`metadata.create_all` 대신 alembic을 쓰는 이유)

`admin/admin` 초기 계정은 스키마 생성 후 lifespan `bootstrap_admin()`에서 생성됨을 확인.

## 구현 단계

1. `backend/entrypoint.sh` 신규 — `alembic upgrade head` 후 `exec "$@"`. 실행권한 부여.
2. `backend/Dockerfile` — `ENTRYPOINT ["/app/entrypoint.sh"]` 추가, 기존 `CMD`(uvicorn) 유지.
3. `docker-compose.yml` — `mcp` 서비스는 read-only이므로 마이그레이션 엔트리포인트를
   타지 않도록 `entrypoint` 명시 오버라이드(api와 동시 마이그레이션 레이스 방지).
4. `backend/tests/test_migrations.py` 신규 — 격리된 임시 DB에서
   `alembic upgrade head`가 (a) 스키마를 생성하고 (b) 재실행 시 멱등(no-op)임을 검증.
5. `README.md` / `instructions/deploy-build.md` 주의사항 동기화.

## 완료 기준

- 빈 DB에 컨테이너만 올리면 스키마 + admin/admin 자동 생성.
- 기존 스키마 DB 재기동 시 충돌 없이 통과.
- 백엔드 테스트 전체 통과(기존 + 신규 마이그레이션 테스트).
